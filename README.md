# DI

Introdução
O presente projeto tem como objetivo o desenvolvimento de um sistema de gerenciamento de pedidos utilizando Java, JDBC e MySQL. A aplicação foi construída seguindo os requisitos propostos pelo desafio integrador, contemplando conceitos de Programação Orientada a Objetos, persistência de dados em banco relacional, processamento assíncrono e boas práticas de Engenharia de Software. A arquitetura escolhida buscou garantir organização, baixo acoplamento entre componentes, facilidade de manutenção e aderência aos princípios fundamentais da orientação a objetos.

Estrutura do Sistema
O sistema foi dividido em camadas, separando a interface de usuário, as regras de negócio e o acesso aos dados. Essa abordagem permite que cada componente possua uma responsabilidade específica, reduzindo a dependência entre as partes do sistema. A camada de interface é composta pelos menus executados no console, responsáveis apenas pela interação com o usuário. Nessa camada são realizadas operações como leitura de dados digitados, exibição de mensagens e navegação pelas funcionalidades do sistema. As regras de negócio ficam concentradas em classes específicas de serviço, responsáveis por validar informações, controlar fluxos de execução e garantir o cumprimento dos requisitos funcionais. O acesso ao banco de dados foi centralizado em classes DAO (Data Access Object), responsáveis por executar todas as consultas SQL utilizando JDBC. Com essa separação, nenhuma classe responsável pelos menus possui comandos SQL ou dependências relacionadas ao banco de dados, atendendo integralmente ao requisito de isolamento da interface.

Modelagem Orientada a Objetos
A modelagem do sistema foi baseada nas entidades identificadas durante a análise dos requisitos: Cliente, Produto, Pedido e ItemPedido. A entidade Cliente armazena informações referentes aos consumidores cadastrados, contendo identificador, nome e endereço eletrônico. A entidade Produto representa os itens comercializados pelo sistema, armazenando nome, preço, quantidade em estoque e categoria. A entidade Pedido representa uma compra realizada por um cliente, contendo informações sobre status e data de criação. Por sua vez, a entidade ItemPedido é responsável por relacionar os produtos pertencentes a cada pedido, armazenando quantidade e preço unitário. Todas as entidades foram desenvolvidas utilizando encapsulamento, com atributos privados e métodos de acesso controlados. Além disso, foram implementadas validações diretamente nos construtores, garantindo que os objetos sejam criados somente em estados válidos. Essa abordagem segue os princípios da Programação Orientada a Objetos e atende às exigências de Object Calisthenics, evitando a criação de objetos incompletos através de múltiplos métodos setters.

Modelagem do Banco de Dados
Para persistência das informações foi utilizado o Sistema Gerenciador de Banco de Dados MySQL. A modelagem relacional foi construída com foco na normalização dos dados e na integridade referencial. O banco é composto pelas tabelas Cliente, Produto, Pedido e ItemPedido. Os relacionamentos foram definidos utilizando chaves estrangeiras, garantindo consistência entre os registros armazenados. O relacionamento entre Cliente e Pedido foi modelado como um relacionamento de um para muitos, permitindo que um cliente possua diversos pedidos. O relacionamento entre Pedido e Produto foi implementado por meio da tabela ItemPedido, responsável por representar a associação entre os dois elementos. Essa estrutura permite armazenar múltiplos produtos em um único pedido, mantendo a flexibilidade necessária para futuras expansões do sistema.

Persistência de Dados
Toda a comunicação com o banco de dados foi realizada utilizando JDBC puro, sem o uso de frameworks de mapeamento objeto-relacional. Essa decisão foi tomada para atender aos requisitos do projeto e proporcionar maior compreensão dos mecanismos internos de persistência. As consultas foram implementadas utilizando PreparedStatement, aumentando a segurança da aplicação e reduzindo riscos relacionados à injeção de comandos SQL. Além disso, foi utilizado o recurso try-with-resources para garantir o fechamento automático de conexões, consultas e demais recursos do banco de dados, evitando vazamentos de memória e problemas de gerenciamento de recursos.

Controle de Estoque e Integridade dos Dados
Uma das principais regras de negócio do sistema consiste em impedir que produtos sejam vendidos sem disponibilidade em estoque. Antes da criação de um pedido, o sistema realiza verificações para confirmar a existência da quantidade solicitada para cada produto. Caso algum item não possua estoque suficiente, a operação é interrompida e o pedido não é registrado. A atualização do estoque ocorre diretamente no banco de dados através de comandos condicionais, garantindo que valores negativos nunca sejam gravados. Essa estratégia aumenta a segurança dos dados e evita inconsistências em cenários de acesso simultâneo.

Processamento Assíncrono
Para atender aos requisitos do projeto, foi implementado um mecanismo de processamento assíncrono baseado em threads. Quando um pedido é concluído pelo usuário, ele é armazenado no banco de dados com o status FILA. Uma thread executada em segundo plano realiza consultas periódicas ao banco buscando pedidos pendentes de processamento. Ao localizar um pedido disponível, a thread altera seu status para PROCESSANDO, realiza a simulação da execução da tarefa e, ao final, atualiza o status para FINALIZADO. Esse modelo permite que a aplicação principal permaneça responsiva e continue atendendo solicitações dos usuários enquanto os pedidos são processados em paralelo.

Gerenciamento das Conexões da Thread
Seguindo os requisitos estabelecidos no projeto, a thread responsável pelo processamento dos pedidos não compartilha conexões com a aplicação principal. A cada ciclo de execução, uma nova conexão com o banco de dados é aberta exclusivamente para a thread. Após a conclusão das operações necessárias, a conexão é encerrada imediatamente. Essa abordagem reduz o acoplamento entre os processos, evita conflitos de concorrência e garante maior estabilidade na execução do sistema. Além disso, falhas ocorridas durante o processamento assíncrono não afetam diretamente o funcionamento do menu principal.

Conclusão
O desenvolvimento do sistema permitiu a aplicação prática dos conceitos estudados ao longo do semestre, envolvendo Programação Orientada a Objetos, modelagem de banco de dados, JDBC, controle transacional e processamento concorrente. A arquitetura adotada proporcionou uma clara separação de responsabilidades entre os componentes, facilitando a manutenção do código e garantindo aderência aos requisitos técnicos definidos para o projeto. O uso de validações nas entidades, persistência segura com JDBC, processamento assíncrono por meio de threads independentes e modelagem relacional normalizada resultou em uma solução robusta, organizada e alinhada às boas práticas de Engenharia de Software.




COMO CONECTAR NO BANCO

Para conectar um projeto Java a um banco de dados MySQL (ou MariaDB), primeiro é necessário garantir que o servidor do banco esteja instalado e em execução. Normalmente isso é feito por meio do XAMPP ou de uma instalação direta do MySQL Server. O serviço precisa estar ativo e rodando na porta padrão 3306, pois é por ela que a aplicação Java irá se comunicar com o banco.
Depois disso, é preciso criar o banco de dados dentro do MySQL. Esse banco será onde as tabelas do sistema vão ficar armazenadas. Também é recomendável criar um usuário específico para a aplicação, em vez de usar o root. Esse usuário recebe permissões apenas no banco necessário, o que evita problemas de segurança e de autenticação.

Em seguida, deve-se baixar o driver JDBC do MySQL, chamado MySQL Connector/J. Esse arquivo é um .jar que permite que o Java “entenda” o MySQL. Ele precisa ser adicionado ao classpath do projeto, ou seja, o Java precisa saber onde esse arquivo está para conseguir usá-lo durante a execução.
Com tudo isso pronto, é criado no código Java uma classe de conexão. Nela é definida a URL do banco, que contém o endereço do servidor, a porta e o nome do banco de dados. Também são informados o usuário e a senha que serão usados na conexão. Dentro dessa classe, utiliza-se o DriverManager para abrir a conexão com o banco. Além disso, o driver JDBC é carregado para garantir que o Java reconheça o MySQL corretamente.

Por fim, ao executar o projeto, o Java utiliza essa configuração para se conectar ao banco. Se todos os dados estiverem corretos e o servidor estiver ativo, a conexão é estabelecida com sucesso e o sistema pode executar operações como inserir, buscar, atualizar e deletar dados no banco.
Em resumo, a conexão entre Java e MySQL depende de quatro pontos principais: o banco estar rodando, o driver JDBC estar configurado, a URL de conexão estar correta e as credenciais de acesso (usuário e senha) estarem válidas.
